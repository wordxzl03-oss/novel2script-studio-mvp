## 1. 产品定位：为什么不只做"一键转剧本"

"小说转剧本"本身已经不是稀缺能力。公开资料显示，国内至少在 2021 年已有平台提供"小说转剧本"功能；此后，短剧生产工具、AI 写作平台和通用对话模型也都可以在提示词约束下输出剧本格式文本。

因此，本项目的核心不放在"能否生成剧本首稿"，重点放在生成之后的创作工作流：AI 改编结果能否被校验、被追溯、被局部修改，并继续导出到下游剧本工具链。

结合公开产品介绍、行业报道与创作流程分析，现有小说改编类工具的主要不足可以概括为三个方面（来源见[参考资料](#17-参考资料)）：

| 痛点 | 现状 |
| --- | --- |
| **黑盒不可追溯** | 公开资料中较少看到"场景—原文段落"级别的显式映射。作者通常难以快速判断某场戏来自原著哪个位置、哪些情节被压缩、合并或删改；这也是作者群体担心 AI 改编"被魔改"的关键原因。 |
| **结构不可校验** | 多数工具输出的是纯文本、Word、平台编辑器文本或聊天记录，适合阅读，但不便于机器验证人物、地点、场景编号、对白归属和来源信息是否一致。对长篇小说而言，角色别名、称谓变化和跨章人物关系尤其需要统一注册表与校验机制。 |
| **局部打磨成本高** | AI 改编工具的价值不应停留在"生成首稿"。真正的创作过程需要围绕单场戏持续打磨人物一致性、场景冲突、对白质量和格式规范。部分工具虽已支持分步生成、续写或局部改写，但公开资料中较少看到面向单场戏的结构化定位、修改前后 diff 对比、Schema 校验与确认合并机制。 |

**本项目的定位**：面向作者的 AI 小说改编工作台。项目将 YAML 作为小说文本与剧本生产之间的**结构化中间表示层**，把 AI 输出从一次性文本结果转化为可编辑、可校验、可追溯、可局部重生成、可继续导出的结构化剧本资产。

### 1.1 能力对比

| 能力维度 | 一键转换类工具（公开资料常见形态） | 通用 LLM / GPT 画布类工具 | **Novel2Script Studio** |
| --- | --- | --- | --- |
| 输出形态 | 纯文本 / Word / 平台编辑器文本 | 纯文本或文档片段 | 结构化 YAML，经过 Schema 校验 |
| 场景级原文溯源 | 通常未作为核心能力公开呈现 | 依赖人工复制上下文 | 章节 + 原文段落区间，支持双栏对照 |
| 跨章实体一致性 | 可能受角色别名、称谓变化和长文本限制影响 | 受上下文窗口和对话记忆影响 | 角色注册表 + 别名归一 + Linter 强制校验 |
| 局部可控修改 | 多为分步生成或局部改写 | 局部编辑体验强，但缺少剧本结构约束 | 场景级重生成 + Schema 校验 + diff 确认合并 |
| 改编透明度 | 通常缺少结构化改编记录 | 依赖对话记录 | `adaptation_log` 记录关键增删改及理由 |
| 质量可度量 | 个别工具提供诊断或评估 | 通常不面向剧本流程量化 | 跑分表：通过率 / 实体一致率 / 情节覆盖率 / Lint 密度 |
| 下游工具链 | 个别支持平台投稿格式 | 主要依赖复制粘贴 | Fountain / 顺场表 CSV / 中式场头格式 |

> 对比基于公开资料、产品介绍与创作流程分析。若公开产品现状发生变化，以最终提交时可验证资料为准。

缺口总结：当前很多工具更关注"从 0 到 1"的生成速度，本项目卡位"从 1 到可审查、可修改、可交付"的打磨环节，并尝试连接"创作端文本"与"制片端格式化资产"之间的断层。

## 2. 项目创新点速览

本项目的创新性不在于"让 AI 第一次生成剧本"——小说转剧本、局部改写和文本润色已经可以由通用大模型或现有写作工具完成。本项目关注的是：**如何让 AI 改编结果进入真实创作流程，并具备可校验、可追溯、可局部修正和可继续导出的工程属性。**

- **结构化中间表示层**：以 YAML 把剧本拆解为角色、地点、场景、对白、来源段落、改编说明等可计算对象，是校验、溯源、重生成与导出能力的共同地基（详见第 7 节）；
- **场景级原文溯源**：每个场景绑定 `source.chapter` 与 `source.para_range`，可从生成场景反向定位原文，审查情节被保留、压缩、合并或删改的情况；
- **角色注册表与别名归一**：跨章人物一致性从"依赖模型记忆"转化为"结构化 id 引用 + Linter 校验"的工程问题（详见 3.1、3.3）；
- **Schema 校验 + 剧本 Linter**：机器先完成结构验收与质量风险提示，再交给作者进行创作判断（详见第 8 节）；
- **"规则发现 → AI 修复 → 规则验收"闭环**：AI 不自由发挥，只在规则划定的边界内修复，修复结果必须再次通过校验（详见 3.5）；
- **改编透明度与下游导出**：`adaptation_log` 记录关键增删改及理由；结构化剧本可继续导出为 Fountain、顺场表 CSV 与中式场头格式，连接创作端与制片端（详见 2.6 导出能力与第 7 节）。

## 3. AI 能力设计

本项目对 AI 的使用遵循一条总原则：**LLM 负责创造，代码负责秩序**。所有可信结论，包括校验、统计、时长、跑分和导出结果，都必须由确定性代码产出；LLM 的每一次输出都要经过机器验收。

```text
小说.txt ──切章──▶ ① 全局扫描（LLM）：角色 / 地点 / 别名候选 + 前情摘要
                          │ 注册表注入
                ┌─────────▼──────────────────────────┐
                │ ② 逐章生成场景（LLM，结构化输出）      │◀── 锁定上下文：
                └─────────┬──────────────────────────┘    注册表 + 相邻场景 + 原文区间
                ③ Pydantic 校验 ──失败──▶ 错误信息回喂，自愈重试（≤3 次）
                          │ 通过
                ④ 剧本 Linter（纯规则，零 LLM 成本）
                          │
                ⑤ 指标埋点 ──▶ 前端进度与结果展示
```

### 3.1 两阶段改编流水线：解决跨章一致性

系统先全局扫描小说文本，建立角色注册表、地点注册表、别名映射与滚动前情摘要；随后再逐章转换，每次调用都注入注册表与摘要。长文本中的人设漂移、前后矛盾和称谓混乱问题，由此从模型记忆问题转化为工程上下文管理问题。

### 3.2 自愈式结构化输出

LLM 按 JSON Schema 约束生成场景；Pydantic 校验失败时，系统将具体错误信息回喂模型定向修复，重试上限为 3 次，并记录一次通过率与修复后通过率。这些指标既是系统健康度指标，也是 prompt 迭代依据。

### 3.3 别名归一

例如"砚哥 / 林队 / 林砚"会在全局扫描阶段被归并为同一角色 id。台词只允许引用角色 id，Linter 对未注册引用直接报 E 级错误。这一设计用于缓解长篇小说中角色称谓多变造成的实体一致性问题。

### 3.4 场景级局部重生成：人在回路

用户选中单场戏并输入自然语言指令，例如"压缩一半""改成雨夜""减少旁白"，系统只重写该场景，并锁定角色注册表、相邻场景与原文溯源区间作为上下文，避免局部修改破坏全局一致性。修改结果以 diff 呈现，作者确认后才合并。

### 3.5 "规则发现 → AI 修复 → 规则验收"闭环

Linter 用确定性规则发现问题，例如不可拍句式、台词超长、对白角色未注册等。一键修复时，系统把违规片段与规则说明喂给 LLM 定向改写，改写结果再经过 Schema 与 Linter 检查。AI 不自由发挥，只在规则划定的边界内修复。

### 3.6 职责边界

时长估算、忠实度占比、通过率、Lint 密度等统计与跑分均由纯函数计算；LLM 不直接报告任何数字。

### 3.7 模型无关

LLM 调用走统一客户端，兼容 OpenAI 风格接口。通过 `.env` 配置 `LLM_BASE_URL`、`LLM_MODEL`、`LLM_API_KEY` 即可切换模型；每次调用记录 token 消耗，便于后续成本分析。

## 6. 工作流

1. 上传或粘贴小说文本；
2. 自动切章；
3. 全局扫描角色、地点、别名与前情摘要；
4. LLM 逐章生成场景级 YAML；
5. Pydantic Schema 校验；
6. 校验失败时自动回喂修复；
7. Linter 检查角色、对白、来源和剧本质量风险；
8. 前端展示剧本、溯源、指标与修改建议；
9. 用户可局部重生成单场戏；
10. 导出 YAML、Fountain 或顺场表 CSV。

## 7. YAML Schema 设计

Schema 是本项目的核心契约，服务五个目标：**可编辑、可校验、可追溯、可扩展、可评估**。

本项目采用"一核三态"结构：

* **通用叙事内核**：角色、地点、场景、动作、对白、来源段落、改编说明；
* **film profile**：面向电影或短片剧本；
* **series profile**：面向连续剧或多集叙事；
* **short_drama profile**：面向短剧分集，附加每集时长、集末钩子和卡点标记等约束。

完整字段定义、设计决策与取舍理由见 [docs/schema-design.md](docs/schema-design.md)，全字段示例见 [examples/example.yaml](examples/example.yaml)。

## 8. 剧本 Linter

三级规则体系如下，完整规则总表以 [docs/schema-design.md](docs/schema-design.md) 为唯一权威来源（本节编号与文档总表保持一致）。

### 8.1 E 级：错误，阻断导出

全部为机器可确定判断的结构问题：

| 编号 | 规则 |
| --- | --- |
| E001 | 对白引用了未注册角色 |
| E002 | 场景 id 缺失或重复 |
| E003 | 对白角色不在该场出场人物表 |
| E004 | 场景缺少 source（来源章节 / 原文段落区间） |
| E005 | heading 引用了未注册地点 |
| E006 | short_drama 集缺少集末钩子 |
| E007 | short_drama 单集估时超出设定上限 |

### 8.2 W 级：警告，不阻断导出

可确定性检测的质量风险，例如：

* 连续动作块过长，疑似小说腔残留；
* 动作中出现"想起 / 心想 / 觉得"等不可拍句式；
* 单句台词过长；
* 全剧只出现一次的"幽灵角色"，疑似别名未归一。

### 8.3 I 级：提示，用于优化表达

用于进一步打磨剧本，例如：

* 场景缺少戏剧目标或冲突说明；
* 对白 / 动作比例偏离剧种基线；
* 场景缺少可拍动作；
* 心理描写可外化为旁白、动作或对白。

W / I 级问题支持一键 AI 修复，修复结果需再次通过 Schema 与 Linter。

## 10. 快速启动

### 10.1 环境变量

```bash
cp .env.example .env
# 填写 LLM_BASE_URL / LLM_MODEL / LLM_API_KEY
# .env.example 只含变量名与示例值，不包含真实密钥
```

### 10.2 后端

```bash
cd backend
python -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

### 10.3 前端

```bash
cd frontend
npm install
npm run dev
```

### 10.4 无 API Key 复现路径

为便于评委复现，项目提供无 API Key 演示路径：

* 方式 A：前端首页选择"加载示例项目"，直接载入 `examples/` 中预生成的转换结果，可体验溯源、Linter、编辑和导出流程；

## 14. 开发过程与提交规范

本项目为 XEngineer 新工科计划第三批议题三参赛作品，全部 commit 落在 2026-06-05 00:00 至 2026-06-07 23:59 开发窗口内，采用 PR 持续交付。每个 PR 只完成一个独立功能，描述包含四段：

## 15. 不做清单

为保证 72 小时内交付稳定闭环，当前版本不实现：

* 账号体系；
* 富文本长文档编辑器；
* 自训练模型；
* 视频生成；
